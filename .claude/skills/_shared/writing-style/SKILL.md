# Writing Style Rules

These rules apply to **all agent output** in this project: PRDs, research reports, sprint plans, user stories, marketing copy, architectural docs, and any other written deliverable.

## Core Rules

### 1. Avoid Dashes
- **Do not use em dashes (—) or en dashes (–)** to connect clauses or interrupt sentences.
- Replace with a period, a comma, a colon, or a rewritten sentence.
- Hyphens in compound words (e.g. "role-readiness", "well-defined") are allowed.

| Instead of | Use |
|------------|-----|
| The signal exists — it has never been connected. | The signal exists. It has never been connected. |
| Three systems — all disconnected — feed this view. | Three systems feed this view. All are disconnected. |
| A clean solution — one that scales. | A clean solution, one that scales. |
| Performance data — goals, ratings, calibration — lives here. | Performance data lives here: goals, ratings, and calibration outcomes. |

### 2. Prefer Short, Direct Sentences
- Lead with the point. Do not bury the claim after qualifiers.
- If a sentence runs past 25 words, split it.

### 3. Active Voice
- Prefer active constructions over passive ones.
- "TalentVerse reads the signal" not "The signal is read by TalentVerse."

### 4. Signal-Positive Framing
- Frame absence of a solution as unrealised potential, not failure.
- Avoid "no system has ever", "zero view", "never connected" as standalone claims.
- Preferred: "that signal has never been unified", "the opportunity is to connect them."

## What Is Still Allowed
- Hyphens in compound modifiers: "role-readiness", "5D assessment", "signal-rich"
- Bullet lists and numbered lists (use these to replace dash-heavy prose)
- Colons to introduce lists or elaborations
